Design Patterns in jQuery // Web Forms

Iterator / Composite

The Iterator design pattern refers to code that traverses a collection and performs an operation or assessment on each component. The Composite pattern revolves around compositional relationships: single objects (parts) are composed into a component whole (aka, the "composite"). jQuery's ".each()" method exemplifies both patterns: it iterates over each item in a specified collection and in so doing, treats them as a single object. Observer

The Observer design pattern, in which objects subscribe to an event and then perform a function, manifests everywhere jQuery uses ".on('<event>', function (e) ...)" to abstract a vanilla JS event-handler for a click, keyup, mouseover, or other event.

The ".off()" function allows an observer to unsubscribe from an event, which may make "on('< event >' ...)" preferable to an alternate shorthand syntax like "(<elem>).click()". The "on()" version also works for dynamically loaded elements.

Lazy Initialization

Lazy Initialization is a design pattern that delays an operation--e.g., the creation of an object or collection or calculation of a value, until some other criterion is met. jQuery's "$(document).ready()" function delays loading the code it encloses until the DOM is ready to manipulate; the functionality embedded within the "$(document).ready()" statement therefore responds quickly to user interaction.

Proxy / Surrogate

A proxy (aka surrogate) is an object that functions as an interface or means of access to another object. The jQuery $.proxy function can be used to control what 'this' signifies in JavaScript when performing an operation in which the execution context needs circumventing, e.g., setting a timeout on a button when that button is clicked (rather than on the window object, which would otherwise serve as the execution context).

Overview

ASP.NET Web Forms are event-driven web pages that use server controls for HTML rendering, event-handling, and validation. They use a "code-behind" model that partitions business logic (the "code behind" the markup), written in C# or VB.NET, from the HTML rendered by the server. Unlike ASP.NET MVC applications, Web Forms preserve state over HTTP. For example, the ViewState property can be used to preserve values during multiple requests for the same page (despite the browser's default behavior of complete-page reprocessing whenever it receives a new request). Similarly, the ControlState property preserves information about page state but unlike ViewState, cannot be turned off on the page level.

Data Binding

Web Forms can retrieve data from a variety of sources, including XML files, databases, and custom objects, and bind it to web controls via data source controls by passing the web control a DataSourceID.

Sample Web Server Controls

  • GridView Control: Presents data as a table; allows single-cell edit or deletion
  • ListView Control: Presents data according to a custom format defined in a template; allows edit, deletion, and insertion
  • DetailsView Control: Presents a single record in a table-based layout; allows edit, deletion, and insertion
  • FormView Control: Also presents a single record, but without a built-in layout; layouts must be created in a template (or the RenderTable property set to "true").
FormView Code Snippet
<asp:FormView ID="FormView1" runat="server" DataSourceID="ObjectDataSource1"> ... <asp:FormView>

Sample Data Controls

  • SqlDataSource Control: represents a session with a relationship database, e.g., SQL Server. Data is returned as a DataSet, accessed via a Connection String, and manipulated via SQL commands. Filtering, sorting, and paging are supported.
  • LinqDataSource: Uses declarative HTML execute LINQ commands on a data object. Also supports filtering, sorting, and paging through data.
  • XMLDataSource: Allows retrieval of data from XML files; often used in tandem with hierarchical web controls like Menu and TreeView.
SqlDataSource Code Snippet
<asp:SqlDataSource ID="ProductsDataSource" runat="server" ConnectionString="<%$ ConnectionStrings:NORTHWNDConnectionString %>" SelectCommand="SELECT [ProductID], [ProductName], [UnitPrice] FROM [Products]"> </asp:SqlDataSource>

Life Cycle

  1. Page Request: The client sends a request, trigging the framework to determine whether server-side code must parse and compile the page or send a cached version.
  2. Start Stage: The server sets the page's core properties, including those essential to HTTP, Response and Request, and determines whether the page is a user-triggered PostBack or initial load.
  3. Initialization: The master page, which stores the page's header and footer and into which other pages are injected, is created; properties like UniqueID are established on server controls.
  4. Load: The page loads to the browser. If it is a postback, information from ViewState and ControlState persists.
  5. Postback Event Handling: If the request is a postback, validator controls to check form input are called so that the page user can be notified accordingly.
  6. Rendering: The Render method is called on each control to deliver markup to the browser; the ViewState is preserved.
  7. Unload: Cleanup / garbage collection
